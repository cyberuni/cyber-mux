---
title: Library API
description: Use cyber-mux as a library — the same cross-multiplexer core the CLI drives, importable directly.
---

Everything the [CLI](/cyber-mux/cli/) does is a thin shell over a pure core, and that core ships as a
library. Import it when you are building a tool *on top of* cyber-mux — a dispatcher, an agent
harness, a workspace manager — rather than shelling out to the `cyber-mux` binary and parsing its
output.

The library is the seam and nothing else: it contains **no** `console.log` and **no** `process.exit`.
Those live only in the CLI modules, which the library entries deliberately do not re-export, so the
process-owning surface is structurally unreachable from every import path below.

## Install

```bash
npm install cyber-mux
```

`cyber-mux` is ESM-only and ships its own types (`.d.mts`). It has one runtime dependency
(`commander`, used by the CLI); the library entries pull in none of it.

## Entry points

Three subpaths, each a different slice of the core. Import from the narrowest one that covers your
need.

| Import | What it is | Page |
| --- | --- | --- |
| `cyber-mux` | The mux core — the [`MuxAdapter`](/cyber-mux/api/mux-adapter/) contract, backend selection, the three adapters, [`nudge`](/cyber-mux/api/nudge/), the [probe](/cyber-mux/api/probe/), and the `Exec`/`NewId` seams. | [MuxAdapter](/cyber-mux/api/mux-adapter/) |
| `cyber-mux/worktree` | The git-worktree adapter and its read helpers — list, resolve, and remove worktrees safely. | [Worktree](/cyber-mux/api/worktree/) |
| `cyber-mux/template` | The template schema, its validator, the flat-N desugarer, and the filesystem-backed resolution seam. | [Template](/cyber-mux/api/template/) |

## The seam pattern

Every core function takes its side effects as parameters rather than reaching for `node:child_process`
or `node:fs` directly. The two you will pass most often:

- **`Exec`** — the synchronous command runner every adapter uses. The real one is `realExec`; a test
  passes a fake that records calls, so an adapter is drivable with no real multiplexer.
- **`NewId`** — the id-minting seam. The real one is `realNewId` (a v4 UUID); a test passes a
  deterministic counter.

Bind the real implementations at the edge of your program and thread them through — that is exactly
what the CLI does.

## Quick start

Detect the backend, open a pane, hand it its turn, and read the result back:

```ts
import { selectMuxAdapter, callerPane, realExec, nudge } from 'cyber-mux'

// Pick the adapter for whatever multiplexer this process is inside.
const adapter = selectMuxAdapter(process.env)

// Open a pane to the right of THIS one, running a command.
const opened = adapter.open(realExec, {
  cwd: process.cwd(),
  at: 'pane:right',
  from: callerPane(adapter, process.env), // split the caller, not the user's focused pane
  launch: 'claude',
})

// Submit a message and verify the peer actually took its turn.
const { taken, resubmits } = await nudge(adapter, realExec, opened, 'run the tests')
console.log({ taken, resubmits })

// Read the pane's current output.
console.log(adapter.read(realExec, opened, { lines: 40 }))
```

`selectMuxAdapter` throws if the process is in no supported multiplexer;
[`probeMultiplexer`](/cyber-mux/api/probe/) is the lower-level call when you want to inspect the
detection result rather than get an adapter.
